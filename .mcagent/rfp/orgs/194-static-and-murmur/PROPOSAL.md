# Static & Murmur — Technical Proposal

**RFP:** `but-ai` Plugin for GitButler
**Approach:** Generative Parameter Exploration for Sonic Installations

---

## Executive Summary

Static & Murmur proposes an agent system designed for aesthetic exploration of parameter spaces. Agents generate multiple configuration variations of sonic installations, each representing a different artistic interpretation of the same RF signal data. The version control system manages parallel creative possibilities as virtual branches, and the human artist selects from the generated space.

---

## Requirement 1: PATH-Based Plugin Architecture

The `but-ai` binary installs to `~/.gitbutler/bin/`. Anton builds it in Rust with real-time audio awareness — the binary can query the installation's rendering pipeline to determine current hardware constraints (available speakers, processing headroom, active frequency captures).

Subcommands:
- `but ai survey` — analyze the RF environment at a venue and produce a frequency map
- `but ai vary` — generate N parameter variations from a base configuration
- `but ai listen` — present variations to the curator with metadata (no audio playback — that happens in the DAW; this shows parameter differences)
- `but ai select` — apply the selected variation and archive alternatives

The `vary` command is the core creative tool. It takes a base installation configuration and produces N variations by exploring the parameter space around it. Each variation becomes a virtual branch. The artist browses branches like browsing sketches.

## Requirement 2: Provider-Agnostic AI

Provider selection is per-task. Creative generation (parameter variations) uses Anthropic for nuanced parameter reasoning. Signal analysis (RF survey interpretation) uses Ollama locally because RF capture data should not leave the studio network.

The provider interface: `generate_variations(base_config, constraints, n) -> Vec<Variation>`. Single method. Each variation includes a `character` field — a short text description of the variation's aesthetic quality, generated by the model ("sparse, rhythmic, with a metallic shimmer"). These descriptions help Zara navigate the variation space without listening to every option.

No fallback chain. If the provider is unavailable, Anton reports it and the collective waits. Art is not urgent.

## Requirement 3: But Agent (INDEX.patch + COMMIT.msg)

The variation workflow:

1. Elif's survey agent analyzes the venue's RF environment
2. Anton dispatches the variation generator with base configuration and constraints
3. The agent produces N variations (default N=5), each as an INDEX.patch modifying the installation configuration
4. Each variation is committed to a virtual branch: `vary/001`, `vary/002`, etc.
5. Zara listens to rendered variations and selects
6. The selected branch is merged to `main`. Rejected branches are archived with Zara's notes

COMMIT.msg format (written by the generation agent, reviewed by Zara):

```
Vary: parameter exploration #3 — sparse rhythmic mapping

Character: sparse, rhythmic, metallic shimmer
Base-Config: installations/schiphol/config-v2.toml
Parameters-Modified: timbre_map, spatial_distribution, temporal_density
Frequency-Focus: 3.5GHz band (5G NR)
Spatial-Layout: 12-channel circle, emphasis on rear channels
Curator-Notes: [pending]
```

Zara fills in `Curator-Notes` after listening. The note becomes part of the permanent record.

## Requirement 4: Polyrepo PR Coordination

Multi-venue installations sometimes span repos (e.g., a traveling exhibition with venue-specific adaptations). PR comments carry aesthetic context:

```
[S&M:sync] schiphol-installation#12 → tokyo-moca-installation
Timbre mapping from Schiphol works well with European 5G NR signals.
Tokyo venue uses different 5G frequency bands (n77 vs n78).
Frequency mapping requires adaptation. Spatial layout requires
complete redesign for gallery dimensions.
```

Forge adapter supports GitHub (current). The collective has no plans to add other forges — they are artists, not infrastructure engineers, and Anton can only maintain so much.

## Requirement 5: Agent Memory in Git Branches

Memory branch: `refs/sm/memory/<installation>`. Memory types:

- **`palette`**: Aesthetic preferences learned from Zara's selections. Maps parameter combinations to Zara's curation decisions. TTL: 1 year. Over time, the agent learns which parameter regions Zara tends to select from.
- **`venue-profile`**: RF environment characteristics and hardware constraints per venue. TTL: until next venue survey.
- **`rejection`**: Variations rejected by Zara, with her notes. TTL: 2 years. Rejections are stored longer than selections because the boundary of "not this" is more informative than the center of "yes."

Memory retrieval uses aesthetic similarity — variations are embedded in a parameter space, and retrieval finds memories with similar parameter profiles. This allows the agent to recall what Zara selected (or rejected) in similar aesthetic neighborhoods.

## Requirement 6: Signed Commits via OpenWallet

Each collective member has an OpenWallet DID. Anton manages key rotation (90-day cycle). The collective adds provenance metadata for exhibition purposes: every signed commit includes the installation name, venue, and the collective's name. This chain enables galleries to verify that a specific configuration was created by Static & Murmur and has not been modified — important for exhibitions that sell installations as artworks.

**Unique insight:** Static & Murmur's variation-branch model treats virtual branches as a creative *possibility space* rather than a *workflow mechanism*. Most version control usage assumes branches will eventually converge — the goal is to merge. In creative practice, the goal is often to diverge — to explore as many possibilities as possible and select later. The `but-ai` architecture supports this inverted usage natively: branches are cheap to create, easy to compare, and the rejected branches persist as documentation of the creative process. This reframes `but-ai` not as a productivity tool but as an exploration tool — a system for managing the space of things that *could be*, not just the thing that *is*.
