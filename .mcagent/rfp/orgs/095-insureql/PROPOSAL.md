# InsureQL -- Technical Proposal

**RFP Response: `but-ai` Plugin for GitButler**

---

## Executive Summary

We build AI products for a living. The `but-ai` plugin is not a research project for us -- it is an extension of our existing infrastructure. We know how to ship AI features weekly, manage LLM costs at scale, and coordinate multi-repo releases. This proposal applies that experience directly.

---

## Requirement 1: PATH-Based Plugin Architecture

Binary on `$PATH`. `cargo binstall but-ai`. The binary embeds a lightweight HTTP health server (localhost only, opt-in via `--health-port`) for integration with CI/CD health checks.

**Startup latency:** Under 100ms cold start. We benchmark this in CI. Regressions fail the build.

---

## Requirement 2: Provider-Agnostic AI

`Completer` trait. Four providers. Our experience managing LLM providers for our product gives us an opinionated provider layer:

**Cost-aware routing:** Each task has a cost ceiling. The provider selector picks the cheapest model that meets the task's quality threshold. Quality thresholds are calibrated from historical patch acceptance rates per model.

**Automatic fallback:** Provider failure triggers automatic retry on the next cheapest viable provider. Latency spikes (>5s response time) trigger preemptive failover before timeout.

---

## Requirement 3: But Agent (INDEX.patch + COMMIT.msg)

Agents produce patches. Patches are validated. Validated patches are committed.

**Our addition: Patch Confidence Score.** Every INDEX.patch ships with a `X-Confidence` trailer in COMMIT.msg (0.0-1.0), generated by the agent's self-assessment of patch quality. Confidence below 0.7 triggers automatic review routing. Confidence below 0.4 triggers human-only review (no auto-merge).

This mirrors our product's confidence scores for underwriting queries. The principle is the same: when the machine is unsure, it says so.

---

## Requirement 4: Polyrepo PR Coordination (Forge-Agnostic)

Forge adapter trait. We operate four repos in production today, so this is not theoretical.

**Release train:** Cross-repo PRs are grouped into release trains. A train is a set of coordinated PRs that must merge together or not at all. The train has a conductor (the initiating agent) who monitors all PRs in the train and triggers a coordinated merge when all are approved.

**Schema:** Train metadata is stored in PR comments with a `<!-- but-ai:train:v1 -->` header containing the train ID, leg index, and status.

---

## Requirement 5: Agent Memory in Git Branches

Memory in `refs/insureql/mem/<agent>/`. Embedding-based retrieval reusing our product's embedding infrastructure.

**Memory schema:**
```json
{
  "key": "query-parser-refactor-pattern",
  "value": "...",
  "embedding": [0.12, -0.34, ...],
  "confidence": 0.89,
  "created": "2026-03-28T10:00:00Z",
  "ttl": 1209600
}
```

**Pre-computed embeddings:** Embeddings are generated at write time and stored alongside the value. Retrieval is a vector dot-product, not an LLM call. This keeps retrieval latency under 50ms and costs zero tokens.

---

## Requirement 6: Signed Commits via OpenWallet

All commits signed. Omar manages the infrastructure. Key rotation integrated into the existing CI/CD pipeline.

**CI integration:** The signing key is provisioned from OpenWallet during CI job startup and revoked at job completion. Ephemeral keys per CI run. No long-lived keys in CI environments.

## Unique Insight: Confidence Scores as Routing Signals

Our underwriting product taught us that AI systems should express uncertainty. A query parser that returns "87% confidence" is more useful than one that returns a bare number, because the underwriter knows how much to trust it.

We apply this to agent patches. Every patch carries a self-assessed confidence score. The score is not just for humans -- it drives automated workflow decisions: high-confidence patches can be auto-merged after automated tests pass; medium-confidence patches route to peer review; low-confidence patches route to senior review or are split into smaller, higher-confidence sub-patches.

In six months of internal use, confidence-routed reviews reduced time-to-merge by 38% for high-confidence patches while increasing review thoroughness for low-confidence patches by 22%. The system sorts itself: easy patches ship fast, hard patches get attention.
