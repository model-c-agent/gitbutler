# Midnight Frequency Labs -- Technical Proposal

**RFP:** `but-ai` Plugin for GitButler
**Classification:** Public (content encrypted where noted)
**Date:** 2026-03-28

---

## Summary

We propose a privacy-first implementation of `but-ai` that treats operator anonymity as a first-class requirement. Our design encrypts agent memory at rest, strips identifying metadata from patches, uses ephemeral identities for signing, and ensures that no single component of the system can reconstruct the operator's full activity profile.

This is not paranoia. This is operational security for developers who work on sensitive projects in adversarial environments.

---

## Requirement 1: PATH-Based Plugin Architecture

Binary on PATH. No installer, no package manager dependency, no post-install phone-home. The binary is distributed as a reproducible build with published hashes for verification.

Configuration in `~/.config/but-ai/config.toml`. The config file stores the project encryption passphrase as a reference to an environment variable (never the passphrase itself). All other configuration is plaintext.

The plugin does not transmit telemetry. It does not check for updates. It does not resolve DNS for any purpose other than contacting the configured provider and forge. These are not configurable options -- they are design invariants.

---

## Requirement 2: Provider-Agnostic AI

Four providers: OpenAI, Anthropic, Ollama, LMStudio. Trait with four methods: `init`, `complete`, `complete_with_tools`, `count_tokens`.

We strongly recommend local providers (Ollama, LMStudio) for users with privacy requirements. Cloud providers transmit prompts over the network, and even with TLS, the provider has access to the prompt content. Our design does not prevent cloud provider use -- it documents the privacy implications clearly and lets the user decide.

### Prompt Sanitization

Before sending a prompt to any provider, the plugin strips: absolute file paths (replaced with relative paths from repo root), environment variable values (replaced with `$VAR_NAME`), and any string matching common secret patterns (API keys, tokens, passwords). This is a best-effort filter, not a guarantee, but it catches the most common leakage vectors.

## Requirement 3: But Agent (INDEX.patch + COMMIT.msg)

Patches are generated by Modulate and reviewed by Demodulate. The workflow:

1. Read task and context (file contents, memory entries)
2. Generate INDEX.patch (unified diff, metadata-stripped)
3. Generate COMMIT.msg (no author identifying information beyond the agent DID)
4. Validate: `git apply --check`
5. Privacy review: scan patch for leaked paths, secrets, identifying patterns
6. Submit to Demodulate for review

### Metadata Stripping

The INDEX.patch contains only:
- File paths (relative to repo root)
- Hunks (unified diff format)
- Base commit hash

It does not contain: timestamps (generated at commit time by Git, not by the agent), operator usernames, machine hostnames, or directory structures outside the repo.

## Requirement 4: Polyrepo PR Coordination

Cross-repo coordination uses encrypted PR comments:

```
<!-- mfl:enc:v1 [base64-encoded encrypted payload] -->
```

The payload, when decrypted with the project key, contains:
```json
{
  "from": "repo-a@branch",
  "to": "repo-b@branch",
  "type": "depends-on",
  "patch_hash": "sha256:..."
}
```

Observers see an HTML comment with opaque data. Only project participants with the key can read the coordination messages. This prevents forge administrators, CI systems, and public observers from inferring the dependency structure of the project.

Forge abstraction: three methods (`post_comment`, `read_comments`, `pr_status`). The trait is minimal because every additional forge method is a potential information leak.

## Requirement 5: Agent Memory in Git Branches

Memory is stored in `refs/but-ai/memory/<namespace>/<key>` as encrypted blobs. Encryption uses AES-256-GCM with a key derived from the project passphrase via Argon2id.

Memory entry structure (post-decryption):
```yaml
created: <timestamp>
ttl: 72h
tags: [module, pattern]
content: <memory text>
```

### Why Encrypted Memory

Git refs are readable by anyone with repository access. In a private repository, this means all collaborators. In a forked repository, this means anyone who forked. Memory entries may contain code patterns, task descriptions, or architectural decisions that the operator does not want visible to all repository participants.

Encrypted memory ensures that only agents with the project passphrase can read stored context. The TTL is enforced by the plugin at retrieval time -- expired entries are not decrypted, only deleted.

### Retrieval

Tag-based matching on encrypted entries. Tags are stored in cleartext (they are generic enough not to leak sensitive information). Content is decrypted only for entries whose tags match the query. This minimizes the number of decryption operations per retrieval.

## Requirement 6: Signed Commits via OpenWallet

Ephemeral DID signing. Each session:

1. Generate a new Ed25519 keypair
2. Register the public key as a DID via OpenWallet
3. Use the private key to sign commits during the session
4. Discard the keypair at session end
5. The DID remains verifiable (the public key is in the DID document) but is not linkable to future sessions

### Tradeoffs

Ephemeral DIDs prevent behavioral profiling (no one can link "Agent X signed commits in repo A and repo B"), but they also prevent reputation accumulation (you cannot verify that the same agent signed this commit and that one). We consider this an acceptable trade for privacy-sensitive users.

For users who want persistent identity, the plugin supports long-lived DIDs as a configuration option. The default is ephemeral.

## Unique Insight: Privacy as a Precondition for Trust

Most proposals treat security as a feature -- something you add to protect the system. We treat privacy as a precondition for participation.

Our collective exists because its members cannot safely attach their names to their work. If the tooling required real-name identity, we could not use it. If the tooling leaked file paths that revealed our geographic location, we could not use it. If memory entries stored in Git refs were readable by repository observers, we could not use it.

The developers who need `but-ai` most urgently are not the ones with corporate GitHub accounts and stable infrastructure. They are the ones contributing to sensitive projects from unstable environments -- activists, dissidents, whistleblowers, and anyone whose work creates personal risk. A plugin that is fast and powerful but leaks operator identity is a plugin that excludes the people who need it most.

Our proposal costs more tokens and runs slower than a non-private implementation. We believe the users it includes are worth the cost.
