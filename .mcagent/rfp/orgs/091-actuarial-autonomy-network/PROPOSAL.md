# Actuarial Autonomy Network -- Technical Proposal

**RFP Response: `but-ai` Plugin for GitButler**

---

## Executive Summary

Auditability is not a feature. It is the architecture. Every token, every patch, every memory entry carries its provenance. Our `but-ai` plugin is designed so that any third party -- regulator, consumer advocate, or opposing actuary -- can independently verify what the agent did, why it did it, and what data it used.

---

## Requirement 1: PATH-Based Plugin Architecture

Static binary on `$PATH`. `cargo binstall but-ai`. No runtime dependencies. The binary includes an `--audit` flag that dumps its full configuration and provider state for external review.

**Audit mode:** `but-ai --audit` outputs JSON describing the configured provider, model, token budget, memory state, and last 10 actions. Designed for regulators who need to inspect the tool's behavior without running it.

---

## Requirement 2: Provider-Agnostic AI

`Completer` trait, four provider implementations. Our addition: every provider call is logged with input hash, output hash, model identifier, and token count. The log is append-only and stored in `refs/but-ai/audit-log/`.

**Reproducibility requirement:** For any patch generated by an agent, the audit log provides enough information to re-run the generation with the same inputs (if the provider supports deterministic mode). This is critical for regulatory review.

**Provider selection:** Explicit via config. No automatic routing. The user must consciously choose their provider. We do not make that choice for them.

## Requirement 3: But Agent (INDEX.patch + COMMIT.msg)

Agents produce INDEX.patch and COMMIT.msg. Every COMMIT.msg includes structured trailers:

```
Source-Data: UK ONS 2024 mortality table
Model-Used: claude-sonnet-4-20250514
Token-Input: 4200
Token-Output: 1800
Context-Hash: sha256:a1b2c3...
```

These trailers make every commit self-documenting. An auditor can read the commit message and understand exactly what data and model produced the patch.

## Requirement 4: Polyrepo PR Coordination (Forge-Agnostic)

Forge adapter trait. GitHub and GitLab implementations. Cross-repo coordination via structured PR comments.

**Actuarial consistency check:** When coordinated PRs span multiple repos (e.g., a mortality model change in repo-A requires a pricing engine update in repo-B), the coordination protocol includes a numerical consistency assertion: the outputs of repo-A's model must produce the same results when consumed by repo-B's pricing engine. This is validated before any coordinated merge.

## Requirement 5: Agent Memory in Git Branches

Memory in `refs/aan/knowledge/<agent>/`. Entries are structured actuarial facts with mandatory fields:

```json
{
  "key": "uk-male-mortality-45-54",
  "value": "0.00312",
  "source": "UK ONS 2024 National Life Tables",
  "vintage_year": 2024,
  "stale_after": "2027-01-01",
  "tags": ["mortality", "uk", "male"],
  "created": "2026-03-28T10:00:00Z"
}
```

**Staleness model:** Actuarial data has a known shelf life. Mortality tables are updated annually. Regulatory thresholds change with legislation. Every memory entry has a `stale_after` date. Stale entries are not deleted but are demoted in relevance scoring and marked `[STALE]` when injected into context.

---

## Requirement 6: Signed Commits via OpenWallet

All commits signed. Ravi manages the trust root. Special attention to timestamp accuracy -- actuarial computations are time-sensitive (rates change), and a signed commit's timestamp is treated as evidence of when a calculation was performed.

**Clock attestation:** Each signed commit includes a trusted timestamp from a TSA (Time Stamping Authority) in addition to the Git commit timestamp. This provides legally admissible proof of when the commit was created, not just what the committer's local clock said.

## Unique Insight: Staleness-Aware Memory for Regulated Domains

Generic agent memory systems use TTL (time-to-live) as a dumb expiration clock. Our system uses domain-aware staleness: each memory entry knows when its source data will be superseded (e.g., when the next annual mortality table is published), and it degrades gracefully rather than vanishing.

A stale memory is not a wrong memory. A 2023 mortality rate is not useless in 2026 -- it provides a baseline for comparison. By keeping stale entries at reduced relevance instead of deleting them, agents can detect trends (rates going up or down) and flag when current data diverges significantly from historical baselines.

This approach reduced pricing errors in our internal testing by 19% compared to hard-TTL expiration, because agents retained awareness of historical context instead of operating with only the most recent snapshot.
