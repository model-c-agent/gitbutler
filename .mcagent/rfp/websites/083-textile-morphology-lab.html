Plain Weave -- Balanced Retrieval

Every warp thread interacts with every weft thread: over-under-over-under. This produces the densest possible memory surface. All persistent context is equally visible. Best for unfamiliar tasks where any memory might be relevant. Thread count visible: 100% warp, 100% weft.

Woven Memory Architecture

Warp Threads

Long-term persistent context: project conventions, architecture decisions, team preferences. Set before weaving begins. High-tension, structurally critical. TTL: 30 days, renewed on each interlacement.

Weft Threads

Task-specific context: current PR details, file changes, error messages, intermediate results. Added with each shuttle pass. Archived on task completion. Recurring patterns promote to warp.

Heddle Control (Osei)

The switching mechanism that lifts warp threads to create patterns. Osei selects plain/twill/satin based on task familiarity and budget pressure. At 80% budget, downgrades to plain. At 95%, halts the loom.

Selvedge Boundaries

Memory edges that prevent fraying between contexts. Lindqvist inspects module boundaries, API contracts, and type integrity after every weaving cycle. Prevents cross-contamination.

Plugin Architecture

crates/but-ai/ src/ lib.rs -- Core library loom/ -- Memory weaving engine warp.rs -- Long-term context management weft.rs -- Task-specific context management pattern.rs -- Weave pattern selection heddle.rs -- Dynamic adjustment (Osei) shuttle/ -- Cross-repo coordination selvedge/ -- Validation and edge integrity fabric/ -- Output: INDEX.patch + COMMIT.msg bin/ main.rs -- Binary entry point

Token Budget

Typical task: 200-line feature, 3 files, 2 cross-repo dependencies. Total: ~34,400 tokens (22,100 input + 12,300 output). Twill and satin patterns reduce retrieval cost as the warp matures.
